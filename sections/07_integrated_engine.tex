\section{Integrated MD Engine}

\subsection{Putting It All Together}

Each of the six modules was built and tested on its own. Now they are connected into a single working simulation. The engine is written as one Python class, \texttt{MDSimulation}, which holds all the simulation state, positions, velocities, forces, and the neighbour list, and runs every module in the right order. Setting the parameters once at the start is enough; they flow automatically to every subroutine that needs them.

\subsection{How the Modules Connect}

The simulation runs in three stages: initialisation (once), equilibration, and production. Figure~\ref{fig:md_flowchart} shows how information moves between the six modules during each stage.

\textbf{Initialisation} is performed only once at the beginning of the simulation. First, Module 2 generates the FCC lattice and determines the simulation box length $L^*$ from the target reduced density $\rho^*$. Next, Module 5 assigns initial velocities by drawing from the Maxwell--Boltzmann distribution and subtracting the centre-of-mass velocity so that the total momentum is exactly zero. Module 4 then builds the first neighbour list using the initial particle positions. Finally, Module 1 calculates the initial Lennard-Jones forces and potential energy using that neighbour list. These forces are required before the first Velocity Verlet step can begin.

\textbf{The equilibration loop} runs for $N_{\text{equil}}$ time steps. At each step, Module 3 advances positions and velocities using the Velocity Verlet integrator, while periodic boundary wrapping from Module 2 is applied during the position update itself using the modulo operation. After the move, Module 4 checks whether any atom has moved more than half the neighbour-list skin distance from its reference position; if so, the neighbour list is rebuilt. Every \texttt{rescale\_interval} steps (10 in this work), Module 5 rescales the velocities so that the system temperature remains close to the target value $T^*_{\text{target}}$. No structural data is collected during equilibration because the system is still relaxing toward equilibrium.

\textbf{The production loop} follows the same integration and neighbour-list procedure, but the thermostat is switched off so that the dynamics evolve naturally in the NVE ensemble. Instead of velocity rescaling, Module 6 samples the radial distribution function $g(r)$ every \texttt{rdf\_interval} steps. These snapshots are accumulated and averaged over time to obtain the final RDF used for structural analysis.

The same sequence is repeated for every temperature point in the melting sweep; only the target reduced temperature $T^*$ changes between runs.

\subsection{The Velocity Verlet Kernel}

The core of both loops is \texttt{\_verlet\_step}, which runs the full four step Velocity Verlet algorithm in four lines:

\begin{lstlisting}[caption={The integration kernel in \texttt{md\_engine.py}. All four lines implement Equations~\eqref{eq:vv_pos} to \eqref{eq:vv_vfull} directly. The \texttt{\% L} on line 2 applies the PBC wrap in the same operation as the position update.}, label={lst:vv_kernel}]
def _verlet_step(self, pos, vel, f, L, nb):
    pos_new = (pos + vel*self.dt + 0.5*f*self.dt**2) % L  # step 1 + PBC
    v_mid   =  vel + 0.5*f*self.dt                         # step 2
    pe, f_new = self._calculate_forces(pos_new, nb, L)     # step 3
    return pos_new, v_mid + 0.5*f_new*self.dt, pe, f_new  # step 4
\end{lstlisting}

Line 2 is worth a closer look: the modulo operator \texttt{\% L} wraps atoms back into the box in the same line as the position update; there is no separate PBC call, and it handles arbitrarily large displacements correctly. Line 4 uses the forces at the \emph{new} positions to complete the velocity update. This is what separates Velocity Verlet from the Euler method and is the reason total energy stays flat over long runs.
\subsection{Main Loop Code}

The listing below shows the complete main loop. Each call is annotated with the module it belongs to.

\begin{lstlisting}[caption={Integrated MD main loop from \texttt{md\_engine.py}. The structure is the same for every temperature step in the melting sweep; only \texttt{T\_star} changes.}, label={lst:main_loop}]
# ===================== INITIALISATION ================================
pos, L    = self._generate_fcc_lattice()            # Module 2
N         = len(pos)
r_max, dr_bin = L / 2.0, (L / 2.0) / self.rdf_bins

T_star    = T_target / self.eps_kb
vel       = self._initialize_velocities(N, T_star)  # Module 5
nb_list   = self._update_neighbor_list(pos, L)      # Module 4
pos_ref   = pos.copy()
pe, force = self._calculate_forces(pos, nb_list, L) # Module 1+2

# ===================== EQUILIBRATION ================================
for step in range(self.equil_steps):
    # Advance one time step (Module 3 + 2)
    pos, vel, pe, force = self._verlet_step(pos, vel, force, L, nb_list)

    # Rebuild neighbour list if any atom moved too far (Module 4)
    if self._needs_rebuild(pos, pos_ref, L):
        nb_list = self._update_neighbor_list(pos, L)
        pos_ref = pos.copy()

    # Rescale velocities to hold temperature (Module 5)
    if step % self.rescale_interval == 0:
        vel = self._rescale_velocities(vel, T_star)

# ===================== PRODUCTION ================================
g_acc = np.zeros(self.rdf_bins)
count = 0

for step in range(self.prod_steps):
    # Advance one time step (Module 3 + 2)
    pos, vel, pe, force = self._verlet_step(pos, vel, force, L, nb_list)

    # Rebuild neighbour list if needed (Module 4)
    if self._needs_rebuild(pos, pos_ref, L):
        nb_list = self._update_neighbor_list(pos, L)
        pos_ref = pos.copy()

    # Sample g(r), thermostat is OFF (Module 6)
    if step % self.rdf_interval == 0:
        _, g_curr = self._calculate_rdf(pos, L, dr_bin, r_max)
        g_acc    += g_curr
        count    += 1

g_avg = g_acc / count   # time averaged RDF
\end{lstlisting}

The structure is short on purpose: the real work happens inside the module methods. What the main loop does is decide \emph{when} to call each one, and in what order.

One detail worth noting is \texttt{pos\_ref}. This stores where every atom was the last time the neighbour list was rebuilt. After each step, \texttt{\_needs\_rebuild} measures how far each atom has drifted from \texttt{pos\_ref}. If any single atom has moved more than $\delta_\text{skin}/2$, the list is rebuilt. This check costs one array operation per step and guarantees no close pair is ever missed between rebuilds.

The skin thickness $r_s^* - r_c^* = 0.5\,\sigma$ means the rebuild threshold is $0.25\,\sigma$. At $T^* = 1.0$ with $\delta t^* = 0.005$, this threshold is crossed roughly every 6 to 8 steps, so the neighbour list is reused for around 85\% of all force evaluations. The $g(r)$ is sampled from 100 frames in the test run and 200 frames in the Kr runs, spread over enough production time for the peaks to converge smoothly.
